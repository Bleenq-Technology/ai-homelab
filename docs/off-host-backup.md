# Off-host backups — jarvis → Asustor NAS

True off-site/off-host disaster-recovery copy of the homelab's nightly datastore
backups. Closes the gap called out in [`docker/backup.sh`](../docker/backup.sh) and the
README roadmap: until now the local dumps **and** the MinIO copy both lived on jarvis's
single disk, so nothing survived loss of that disk.

## Architecture

```
jarvis (Ubuntu)                              stor.pdx.sanctioned.tech  (Asustor AS6706F / FS6706T, x86_64)
─────────────────                            ──────────────────────────────────────────────────────────
/opt/homelab/backup.sh   (cron 0 2 * * *)    ADM 5.1.3 · Docker 28.x · App Central "docker-ce"
  ├─ pg_dumpall / CH BACKUP / QuestDB / MinIO   ┌──────────────────────────────────────────────┐
  └─ backup_offhost():  restic backup  ───────► │ restic/rest-server  --private-repos            │
        $BK = /opt/homelab/backups               │                     --append-only :8000 (HTTP) │
        RESTIC_REPOSITORY=rest:http://jarvis:***@stor:8000/jarvis/   │ repo data → /volume1/Docker/restic-rest/data/jarvis
        RESTIC_PASSWORD=*** (repo encryption)    └──────────────────────────────────────────────┘
                                                  repo data → /volume1/restic-backups/jarvis (shared folder)
```

- **restic** does client-side **encryption** (the NAS never sees plaintext — important:
  the dumps include the Infisical/Keycloak/app Postgres data), **deduplication** (nightly
  near-identical dumps cost ~0 B after the first), and **integrity verification**.
- **`--append-only`**: jarvis can *add* snapshots but cannot *delete* them. A compromised
  or ransomwared jarvis cannot wipe backup history (verified: a delete returns `403`).
- **`--private-repos`**: the `jarvis` account is scoped to the repo path `/jarvis`.
- Transport is plain **HTTP on the trusted LAN/VLAN**. The repo contents are encrypted by
  restic regardless; HTTP only carries the basic-auth credential + already-encrypted blobs.
  *Hardening follow-up:* rest-server supports `--tls` (self-signed) if desired.

## Credentials (all in Infisical `homelab/prod`)

| Var | Role |
|-----|------|
| `ASUSTOR_HOST_NAME` | NAS hostname — `stor.pdx.sanctioned.tech` (resolves to `192.168.2.11`) |
| `ASUSTOR_INTEGRATION_LOGIN` | NAS account **and** rest-server basic-auth user — `jarvis` |
| `ASUSTOR_INTEGRATION_PASSWORD` | the NAS `jarvis` account password (SSH / SMB / sudo) |
| `RESTIC_REST_PASSWORD` | rest-server basic-auth password for user `jarvis` (URL-safe hex) |
| `RESTIC_PASSWORD` | **repo encryption key** — losing it makes every backup unrecoverable |

> `RESTIC_PASSWORD` belongs in the **bootstrap set** vaulted in a password manager
> (alongside `INFISICAL_ENCRYPTION_KEY`): if jarvis *and* Infisical are both lost, you
> still need this key to read the NAS backups during a rebuild. See
> [`core/infisical/README.md`](../docker/core/infisical/README.md).

`backup.sh` reads them from `/opt/homelab/.env` (regenerated by `pull-secrets.sh`). To
rotate any of them: set in Infisical (`./push-secret.sh KEY VALUE`) → `./pull-secrets.sh`.
The rest-server user password (`RESTIC_REST_PASSWORD`) is *also* baked into the NAS
`.htpasswd` (bcrypt) — rotating it means regenerating that file too (see below).

## The rest-server container (on the NAS)

Defined at `/volume1/Docker/restic-rest/compose.yml` (managed over SSH; the NAS is **not**
a git checkout):

```yaml
services:
  restic-rest:
    image: restic/rest-server:0.13.0@sha256:8668c235a932745585fef5cabbab9eccb501035f90138085a6e74963dcd1a71f
    container_name: restic-rest
    restart: unless-stopped
    environment:
      - OPTIONS=--private-repos --append-only
    ports:
      - "8000:8000"
    volumes:
      - /volume1/restic-backups:/data               # dedicated shared folder; repo + .htpasswd live here
```

- **Auth gotcha:** do **not** set `DISABLE_AUTHENTICATION` — the image disables auth if that
  var is set to *any non-empty value* (even `false`). Omitting it keeps `.htpasswd` auth on.
- The `restic-backups` shared folder (mounted at `/data`) holds `.htpasswd` (user `jarvis`,
  bcrypt) and the repo at `jarvis/`. The compose file itself lives at `/volume1/Docker/restic-rest/`.
- ADM requires the `jarvis` account to be in the **administrators** group for SSH/Docker
  management. The *backup path itself* only needs rest-server HTTP auth — so once deployed,
  `jarvis` could be dropped from administrators / Terminal disabled and backups still run.

Regenerate `.htpasswd` (e.g. after rotating `RESTIC_REST_PASSWORD`), run on jarvis:

```bash
cd /opt/homelab
RRP=$(grep ^RESTIC_REST_PASSWORD= .env | cut -d= -f2- | tr -d "'\"")
htpasswd -nbB jarvis "$RRP" | ssh jarvis@stor.pdx.sanctioned.tech 'cat > /volume1/restic-backups/.htpasswd'
```

## Restore runbook

On any host with `restic` + the two repo secrets:

```bash
export RESTIC_REPOSITORY="rest:http://jarvis:<RESTIC_REST_PASSWORD>@stor.pdx.sanctioned.tech:8000/jarvis/"
export RESTIC_PASSWORD="<RESTIC_PASSWORD>"

restic snapshots                              # list available points in time
restic restore latest --target /tmp/restore   # restore the newest snapshot
restic restore <ID> --target /tmp/restore --include /opt/homelab/backups/postgres-all-XXXX.sql.gz
```

Then restore the chosen dump into the live service the normal way (e.g. Postgres:
`gunzip -c postgres-all-*.sql.gz | docker exec -i postgres psql -U postgres`). A full
DR rebuild also needs the Infisical Postgres DB + `INFISICAL_ENCRYPTION_KEY` — see the
Infisical README "Rebuild from scratch".

## Retention / pruning (deliberate maintenance — append-only)

`--append-only` blocks `restic forget`/`prune` from jarvis **by design** (that's the
tamper-resistance). Data is tiny and deduplicated, so the repo grows slowly; prune only
occasionally, as a deliberate maintenance window:

```bash
# on the NAS, temporarily relax append-only:
sed -i 's/ --append-only//' /volume1/Docker/restic-rest/compose.yml
sudo docker compose -f /volume1/Docker/restic-rest/compose.yml up -d
# then from jarvis, apply a retention policy + prune:
restic forget --keep-daily 7 --keep-weekly 4 --keep-monthly 6 --prune
# re-enable append-only (restore the flag) and recreate:
#   OPTIONS=--private-repos --append-only  → docker compose up -d
```

## Operational notes

- No new cron — the off-host step rides the existing nightly `backup.sh` (`0 2 * * *`).
  Failures log `[ERROR] off-host (restic->NAS) FAILED` and (like every datastore) do not
  abort the other backups.
- Sundays the job also runs `restic check` against the repo.
- restic on jarvis: installed via apt then `restic self-update` (0.18.1+).
